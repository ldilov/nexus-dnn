# Nexus DNN — API & Functionality Reference

> **Status:** Living document. Last reconciled against source: `2026-04-28` against branch `035-extension-dependency-installer` (HEAD `30ed192`).
> **Companion file:** [`openapi.yaml`](./openapi.yaml) — machine-readable OpenAPI 3.1 spec covering the same surface.
> **Regeneration:** `cargo run -p nexus-api --bin api-doc-check` validates that this file and the OpenAPI spec are in sync with the live router (see the [Regeneration](#regeneration) section).

This document describes **everything a client can do** against a running `nexus-dnn` host:

1. The HTTP surface exposed by the host crate (`crates/nexus-api`).
2. The routes contributed by built-in extensions, mounted under the generic extension router. This document currently gives the deepest route-level detail for `local-llm` and `emotion-tts`, while the repo also ships newer built-ins such as `nexus-video-ltx23`, `nexus-video-longcat`, and `nexus-video-svi2-pro`.
3. The WebSocket and Server-Sent-Events streams.
4. The supporting functionality (deployments, runs, model store, backend runtimes, etc.) — what each surface enables, not just the wire shape.

For the full architectural picture (crate map, data flow, host ↔ extension boundary), read [`architecture.md`](../architecture.md) first.

## Conventions

### Base URL & versioning

| Prefix          | Purpose                                                                |
| --------------- | ---------------------------------------------------------------------- |
| `/api/v1`       | Primary, versioned host + extension surface.                           |
| `/api/host`     | Host introspection (Spec 028) — facts the host exposes to extensions. |
| `/`             | SPA frontend fallback (`apps/web` static bundle).                      |

The host currently binds to `0.0.0.0` by default and has no built-in authentication; trust the local user. In normal local usage, clients should still connect through `127.0.0.1`. If you expose the host beyond loopback, front it with an authenticating reverse proxy.

### Response envelope

All JSON responses, success and error, are wrapped in a standard envelope (defined in `crates/nexus-api/src/dto/envelope.rs`):

```jsonc
{
  "data":  { /* T on success, null on error */ },
  "meta":  { "timestamp": "2026-04-28T12:34:56Z" },
  "error": null  // or an error object on failure
}
```

Error shape:

```jsonc
{
  "data":  null,
  "meta":  { "timestamp": "..." },
  "error": {
    "code":     "NOT_FOUND",            // machine-readable
    "category": "not_found",            // bucket: validation | not_found | conflict | internal | unauthorized
    "message":  "Run xyz does not exist.",
    "details":  { /* optional structured payload */ }
  }
}
```

A handful of routes deviate from the envelope and stream their payload directly (SSE, WebSockets, raw artifact blobs, ZIP downloads). Each such route is flagged in this document.

### Content type & encoding

- JSON unless noted (multipart for voice-asset upload, `application/zip` for export download, `application/octet-stream` for artifact blobs, `text/event-stream` for SSE, WebSocket frames otherwise).
- All timestamps are RFC 3339 UTC strings.
- Identifiers (`run_id`, `deployment_id`, `voice_asset_id`, …) are opaque strings — usually ULIDs — generated by the host or extension.

### Naming

DTO field naming is **mostly snake_case** at the host boundary, but some extension surfaces emit camelCase (notably `nexus.audio.emotiontts/runs/*` and the SSE event payloads — see §[Builtin extension: nexus.audio.emotiontts](#builtin-extension-nexusaudioemotiontts)). The convention per route is documented inline.

### Error categories

| Category       | HTTP status | Typical cause                                                   |
| -------------- | ----------- | --------------------------------------------------------------- |
| `validation`   | 400         | Body or query parameter failed schema validation.               |
| `unauthorized` | 401 / 403   | Reserved for future auth layer; not currently emitted.          |
| `not_found`    | 404         | Resource does not exist.                                        |
| `conflict`     | 409         | Business invariant violated (duplicate name, illegal state).    |
| `internal`     | 500         | Bug, unhandled IO error, dependency outage.                     |

### Deprecation headers

Legacy paths emit RFC 8594 deprecation headers:

```
Deprecation: true
Sunset: <RFC 7231 date>
Link: </api/v1/<replacement>>; rel="successor-version"
```

Two surfaces are currently deprecated:

| Old prefix              | Sunset             | Replacement                              |
| ----------------------- | ------------------ | ---------------------------------------- |
| `/api/v1/llm/backends`  | 2026-05-13 00:00 GMT | `/api/v1/backends` + `/api/v1/host-models` |
| `/api/v1/huggingface`   | 2026-07-19 00:00 GMT | `/api/v1/model-store/search`             |

### Global middleware

Configured in `crates/nexus-api/src/router.rs`:

- CORS — `Access-Control-Allow-Origin: *` for all methods/headers (loopback-only by default).
- Compression — gzip + brotli (Fastest quality).
- Panic catching via `tower_http::catch_panic::CatchPanicLayer` — never crashes the server on a panic.
- Request tracing — TRACE for all, ERROR for 5xx.
- Body limit on the install-from-zip route — 64 MiB.

## Host functionality at a glance

The host process is a **generic runtime**. The big nouns it exposes are:

| Noun                  | What it is                                                                  | Owns persistence?                       |
| --------------------- | --------------------------------------------------------------------------- | --------------------------------------- |
| **Extension**         | An installed plugin (builtin or external) declared via `manifest.yaml`.    | Host (registry); extensions own their tables under `ext_<id>__*`. |
| **Operator**          | A workflow node type contributed by an extension.                          | Host (in-memory registry).              |
| **Recipe**            | A canonical workflow template contributed by an extension.                 | Host.                                   |
| **Workflow**          | A user-authored DAG (graph of operator nodes) — may be edited via canvas. | Host.                                   |
| **Module**            | An aggregated view across extensions, recipes, workflows, deployments.     | Host.                                   |
| **Deployment**        | A configured, runnable instance of a recipe/workflow.                      | Host (cross-extension); extensions may keep deployment-scoped state in their own tables. |
| **Run**               | A single execution of a deployment or ad-hoc workflow.                     | Host (host-level runs); extensions keep per-extension run rows. |
| **Artifact**          | A produced output (file, blob, JSON record) referenced by content hash.    | Host.                                   |
| **Backend runtime**   | A managed binary backend (e.g., llama.cpp, IndexTTS-2 worker).             | Host (`backend-runtimes`, `backend-runtime-installs`, `backend-runtime-leases`). |
| **Host model**        | A model registered with the host registry (post-Spec 031 generic shape).   | Host.                                   |
| **Model store entry** | A catalog item searchable via Spec 025 model-search.                       | Host.                                   |

Extensions always reach the host through one of:

- A typed Rust trait (`HostArtifactStore`, `LeaseProvider`, `ExtensionStorage`, `EventBus`, …).
- A manifest declaration that the host indexes (operators, recipes, UI contributions).
- An HTTP endpoint exposed by the extension and **mounted by the host** under `/api/v1/extensions/:ext_id/*`.

The host never imports an extension's types or knows what a given extension does. See [`docs/extension-internals.md`](../extension-internals.md) and [`.claude/rules/host-extension-boundary.md`](../../.claude/rules/host-extension-boundary.md) for the non-negotiable boundary rules.

## Extensions (lifecycle)

Source: `crates/nexus-api/src/handlers/extensions.rs`. The install-from-zip handler lives in `crates/nexus-api/src/handlers/extensions_install/zip_handler.rs`.

| Method | Path                                          | Description |
| ------ | --------------------------------------------- | ----------- |
| GET    | `/api/v1/extensions`                          | List all discovered extensions with registry status. |
| GET    | `/api/v1/extensions/{id}`                     | Full record for one extension (manifest fields, capabilities, dependency state). |
| POST   | `/api/v1/extensions/refresh`                  | Re-discover extensions on disk and revalidate manifests. |
| POST   | `/api/v1/extensions/{id}/reveal`              | Open the extension's folder in the OS file explorer. |
| POST   | `/api/v1/extensions/{id}/enable`              | Enable; checks declared dependencies and emits `extension_activated`. |
| POST   | `/api/v1/extensions/{id}/disable`             | Disable; releases leases and unmounts router. |
| POST   | `/api/v1/extensions/install-from-zip`         | Install from a zip body (max 64 MiB; FR-IE03 of Spec 019). |

`GET /api/v1/extensions` returns extensions with two complementary fields:

- `status` — frozen state at startup: `active`, `disabled`, `error`.
- `registry_state` — live state: `not_registered`, `registered`, `error`. Extensions whose storage migrations or router mount failed surface as `registered=false`.

## Extension dependencies (Spec 035)

Source: `crates/nexus-api/src/handlers/extension_dependencies/`.

The dependency installer is a typed step graph. An extension's manifest declares `dependencies.steps[]` with handlers like `runtime`, `package_set`, `system_binary`, `model_artifact`, `validation`. The host topo-sorts and walks the plan, broadcasting progress as SSE.

| Method | Path                                                                         | Description |
| ------ | ---------------------------------------------------------------------------- | ----------- |
| GET    | `/api/v1/extensions/{id}/dependencies`                                       | List dependency steps with current status (`pending`, `running`, `succeeded`, `failed`, `skipped`). |
| POST   | `/api/v1/extensions/{id}/install`                                            | Start the install plan. Returns the install id; progress goes through SSE. |
| POST   | `/api/v1/extensions/{id}/install/steps/{step_id}/retry`                      | Retry a failed step in place. |
| POST   | `/api/v1/extensions/{id}/install/cancel`                                     | Cancel the in-flight install. |

Live progress is emitted on the host event bus under topic `extension.install` and surfaced via the WebSocket `/api/v1/events` stream. Five event variants are emitted (defined in `crates/nexus-extension-deps/src/types.rs::ProgressEvent`): `step_started`, `step_progress`, `step_completed`, `step_failed`, `install_completed`.

## Extension storage contributions

Source: `crates/nexus-api/src/handlers/storage_contributions.rs`.

Extensions don't write to the host database directly. They declare their schema in `manifest.yaml` and the host applies it under a `ext_<extension_id_slug>__<table>` namespace. See `docs/database-schema.md` and the host-extension-boundary rule.

| Method | Path                                                  | Description |
| ------ | ----------------------------------------------------- | ----------- |
| GET    | `/api/v1/extensions/{id}/storage`                     | Status of the extension's storage namespace (applied migrations, drift). |
| POST   | `/api/v1/extensions/{id}/storage/validate`            | Dry-run the contribution plan; reports missing tables, column drift. |
| POST   | `/api/v1/extensions/{id}/storage/apply`               | Apply pending migrations. |
| POST   | `/api/v1/extensions/{id}/storage/verify`              | Verify integrity (checksums, FK consistency). |
| POST   | `/api/v1/extensions/{id}/storage/uninstall`           | Drop the namespace; **destructive** — extension data is lost. |
| GET    | `/api/v1/storage/namespaces`                          | List all currently registered storage namespaces. |

## Generic extension router

Source: `crates/nexus-api/src/extension_router/`.

The host exposes a generic mount point that forwards everything after the extension id to the extension's own router. The host code never parses the sub-path.

```
/api/v1/extensions/{ext_id}/                 → extension's root handler
/api/v1/extensions/{ext_id}/{*rest}          → arbitrary sub-paths
```

Supported methods: `GET`, `POST`, `PUT`, `PATCH`, `DELETE`. The registry of extension routers is sealed at startup; runtime registration is not allowed once the HTTP listener is bound.

Built-in extensions live behind this mount point. Two especially mature route surfaces documented in detail below are:

- `nexus.local-llm` — see [§ Builtin extension: nexus.local-llm](#builtin-extension-nexuslocal-llm).
- `nexus.audio.emotiontts` — see [§ Builtin extension: nexus.audio.emotiontts](#builtin-extension-nexusaudioemotiontts).

## Draft AI suggestions (Spec 037 / Phase 8)

Source: `crates/nexus-api/src/handlers/draft_suggestions/`.

Host-owned, **extension-agnostic** stream of inline AI suggestions for Module Drafts. The host acquires an inference backend via the existing `/api/v1/backend-runtime-leases` machinery; the path contains zero extension-id literals (Constitution Principle XIII / FR-051). The wire-format SSE event contract lives at `specs/037-spectral-graphite-redesign/contracts/draft_suggestions.events.md`.

| Method | Path                                                                              | Description |
| ------ | --------------------------------------------------------------------------------- | ----------- |
| POST   | `/api/v1/modules/drafts/{draft_id}/suggestions`                                   | Open an SSE stream of suggestion deltas for one draft line. First event is `stream_started`; tokens follow; terminal event is `complete \| error \| cancelled`. Pre-stream lease failure returns HTTP 503 with `{ code: "no_backend_leasable", cta: { label, href: "/backends" } }`. Validation failure returns 400 with a typed `validation_error` body. |
| POST   | `/api/v1/modules/drafts/{draft_id}/suggestions/{stream_id}/cancel`                | Cancel an in-flight stream (idempotent — always 204, even for unknown / finished `stream_id`). The streaming task observes the cancel flag between tokens and emits a final `cancelled { reason: "client_cancelled" }` event before closing. |

**SSE event types**: `stream_started`, `token`, `partial`, `complete`, `error`, `cancelled`. Each event has an explicit `event:` field — consumers dispatch by type, not by structural payload match. See the events contract for full schemas.

**Boundary audit**: `crates/nexus-api/tests/draft_suggestions/boundary_audit_test.rs` greps the handler module for any extension-id literal on every `cargo test` run.

#### SSE event schema (`/runs/{run_id}/progress`)

`Content-Type: text/event-stream`. The SSE handler retries channel subscription up to 3000 × 100 ms (5-min budget) and DB-polls every ~1 s. On late subscribe, it replays `segment_started` + terminal-status events for already-processed utterances before joining the live channel. Keep-alive comment frames every 15 s.

```jsonc
// segment_started
{ "type": "segment_started", "run_id": "...", "utterance_id": "...", "global_index": 0 }

// segment_completed
{ "type": "segment_completed", "run_id": "...", "utterance_id": "...", "global_index": 0, "duration_ms": 1234 }

// segment_failed
{ "type": "segment_failed", "run_id": "...", "utterance_id": "...", "global_index": 0, "failure_category": "...", "failure_detail": "..." }

// run_terminal
{ "type": "run_terminal", "run_id": "...", "status": "completed|failed|cancelled|partial" }
```

## Regeneration

The host crate ships a small CLI binary that walks the live router and **fails the build** if it finds drift between the live route table and what this document declares.

```sh
# Verify (CI-friendly; non-zero exit on drift)
cargo run -p nexus-api --bin api-doc-check

# Print the live route table (for spotting new endpoints)
cargo run -p nexus-api --bin api-doc-check -- --dump
```

The binary is intentionally small — it walks the same `build()` function the server uses, then matches every route to a checklist entry in `docs/api/openapi.yaml`. New routes show up as drift, fail the check, and force the author to update both this document and the OpenAPI spec.

> **Note (2026-04-28):** the `api-doc-check` bin is being added in the same PR that introduces this doc. Until that lands, regenerate by reading `crates/nexus-api/src/router.rs` directly.
